 Pull requests and contributions are warmly welcome. Please follow existing code style and commit message conventions. Also remember to keep documentation updated.

{\bfseries{Pull requests\+:}} You don\textquotesingle{}t need to bump version numbers or modify anything related to releasing. That stuff is fully automated, just write the functionality.\hypertarget{md_public_2_muse_2assets_2vendor_2progressbar_2docs_2contributing_autotoc_md190}{}\doxysection{\texorpdfstring{Get started with documentation}{Get started with documentation}}\label{md_public_2_muse_2assets_2vendor_2progressbar_2docs_2contributing_autotoc_md190}
Documentation is written in markdown and \href{https://github.com/mkdocs/mkdocs}{\texttt{ mkdocs}} is used to build documentation site. \href{https://readthedocs.org/projects/progressbarjs/}{\texttt{ Read\+The\+Docs}} provides hosting for the documentation site and builds the documentation on each commit.

To locally develop documentation, you need to install mkdocs and requirements specified in pip-\/requirements.\+txt. After that you can run {\ttfamily mkdocs serve}.\hypertarget{md_public_2_muse_2assets_2vendor_2progressbar_2docs_2contributing_autotoc_md191}{}\doxysection{\texorpdfstring{Get started with development}{Get started with development}}\label{md_public_2_muse_2assets_2vendor_2progressbar_2docs_2contributing_autotoc_md191}

\begin{DoxyItemize}
\item Install local environment.
\item {\ttfamily cd local-\/dev} and serve folder to browser
\item Open another terminal to project root and run {\ttfamily grunt watch}.
\end{DoxyItemize}

Then you can edit {\ttfamily src/progressbar.\+js} and changes can be tested in browser. Edit {\ttfamily local-\/dev/main.\+js} to your testing needs.

Shorter way to do local development is running\+: {\ttfamily npm run dev}.\hypertarget{md_public_2_muse_2assets_2vendor_2progressbar_2docs_2contributing_autotoc_md192}{}\doxysection{\texorpdfstring{General project stuff}{General project stuff}}\label{md_public_2_muse_2assets_2vendor_2progressbar_2docs_2contributing_autotoc_md192}
This package uses npm/node tools just in the developer environment. Grunt is used as a task runner but there\textquotesingle{}s no reason it couldn\textquotesingle{}t be replaced with e.\+g. Makefile

{\itshape Progress\+Bar.\+js} depends on tweening library called \href{https://github.com/jeremyckahn/shifty}{\texttt{ shifty}}. {\itshape Shifty} is bundled inside the scripts in \href{https://github.com/kimmobrunfeldt/progressbar.js/blob/master/dist}{\texttt{ dist/}} directory. Dependency is bundled in to ease using the library.\hypertarget{md_public_2_muse_2assets_2vendor_2progressbar_2docs_2contributing_autotoc_md193}{}\doxysubsubsection{\texorpdfstring{Versioning}{Versioning}}\label{md_public_2_muse_2assets_2vendor_2progressbar_2docs_2contributing_autotoc_md193}
Versioning follows \href{http://semver.org/}{\texttt{ Semantic Versioning 2.\+0.\+0}}. The release script makes sure that for each release, there exists only one commit in history where version number in {\itshape bower.\+json} matches the release\textquotesingle{}s version. That commit is tagged as the release, for example {\ttfamily 0.\+4.\+1}. Commits after that have -\/dev suffix({\itshape 0.\+4.\+1-\/dev}) in the version number to avoid any possible confusion.

In other words, if you look into {\itshape bower.\+json}, you can tell if the code base is a released version or not.\hypertarget{md_public_2_muse_2assets_2vendor_2progressbar_2docs_2contributing_autotoc_md194}{}\doxysection{\texorpdfstring{\{install-\/environment\} Install environment}{\{install-\/environment\} Install environment}}\label{md_public_2_muse_2assets_2vendor_2progressbar_2docs_2contributing_autotoc_md194}
Install tools needed for development\+: \begin{DoxyVerb}npm install
npm install -g watchify
npm install -g browserify
\end{DoxyVerb}
 For testing\+: \begin{DoxyVerb}npm install -g mocha
npm install -g testem
\end{DoxyVerb}
\hypertarget{md_public_2_muse_2assets_2vendor_2progressbar_2docs_2contributing_autotoc_md195}{}\doxysection{\texorpdfstring{Test}{Test}}\label{md_public_2_muse_2assets_2vendor_2progressbar_2docs_2contributing_autotoc_md195}
Tests are written with \href{http://mochajs.org/}{\texttt{ Mocha}} + \href{https://github.com/LearnBoost/expect.js/}{\texttt{ expect.\+js}}. Sometimes the tests fail even though they actually work on the browser. That might be cause of set\+Timeouts used in tests.

Quickly run tests\+: \begin{DoxyVerb}grunt test
\end{DoxyVerb}
 This will use testem to run tests with Chrome. Other options specified below.\hypertarget{md_public_2_muse_2assets_2vendor_2progressbar_2docs_2contributing_autotoc_md196}{}\doxysubsubsection{\texorpdfstring{Testem}{Testem}}\label{md_public_2_muse_2assets_2vendor_2progressbar_2docs_2contributing_autotoc_md196}
\href{https://github.com/airportyh/testem}{\texttt{ Testem}} is used for running tests locally. It is fast and easy to use. List of example commands\+:


\begin{DoxyItemize}
\item {\ttfamily testem} Serves testing page so that you can connect any browser to it.
\item {\ttfamily testem ci} Runs tests an all available/detected local browsers.
\item {\ttfamily testem ci -\/R dot -\/l chrome} Runs tests with Chrome using dot reported.
\end{DoxyItemize}\hypertarget{md_public_2_muse_2assets_2vendor_2progressbar_2docs_2contributing_autotoc_md197}{}\doxysubsubsection{\texorpdfstring{Karma}{Karma}}\label{md_public_2_muse_2assets_2vendor_2progressbar_2docs_2contributing_autotoc_md197}
\href{http://karma-runner.github.io/}{\texttt{ Karma}} is used for running tests to be executed in Sauce Labs. Karma was used because integrating it to Sauce Labs is easier than with Testem.

You must setup {\ttfamily SAUCE\+\_\+\+USERNAME} and {\ttfamily SAUCE\+\_\+\+ACCESS\+\_\+\+KEY} environment variables. See \href{https://github.com/karma-runner/karma-sauce-launcher\#username}{\texttt{ karma-\/sauce-\/launcher project documentation}}. Browsers used to test are specified in \href{karma.conf.js}{\texttt{ karma.\+conf.\+js}}.

You can run tests with all browsers\+: \begin{DoxyVerb}grunt karma
\end{DoxyVerb}
 Tests are run sequentially in batches of browsers to prevent timeouts in Sauce Labs. You can also run single set of browsers to see test results faster\+: \begin{DoxyVerb}grunt karma:sauce0
\end{DoxyVerb}
\hypertarget{md_public_2_muse_2assets_2vendor_2progressbar_2docs_2contributing_autotoc_md198}{}\doxysection{\texorpdfstring{Release}{Release}}\label{md_public_2_muse_2assets_2vendor_2progressbar_2docs_2contributing_autotoc_md198}
{\bfseries{Before releasing, make sure there are no uncommitted files, tests pass and jshint gives no errors.}}

Creating a new release of the package is simple\+:


\begin{DoxyEnumerate}
\item Commit and push all changes
\item Run local tests and linters with {\ttfamily npm test}
\item Make sure Sauce Labs tests pass
\item Run {\ttfamily grunt release}, which will create new tag and publish code to Git\+Hub

Bower detects your new version of git tag.
\item Edit Git\+Hub release notes
\end{DoxyEnumerate}

By default, patch release is done. You can specify the version bump as a parameter\+: \begin{DoxyVerb}grunt release:major
\end{DoxyVerb}
 Valid version bump values\+: {\ttfamily major}, {\ttfamily minor}, {\ttfamily patch}.\hypertarget{md_public_2_muse_2assets_2vendor_2progressbar_2docs_2contributing_autotoc_md199}{}\doxysection{\texorpdfstring{Decision log}{Decision log}}\label{md_public_2_muse_2assets_2vendor_2progressbar_2docs_2contributing_autotoc_md199}

\begin{DoxyItemize}
\item Animate SVG paths with CSS3 transitions to make animations smooth.
\item API must provide built-\/in shapes and a way to use totally custom SVG.
\item Document manually. More overhead and risk of out dated information but easier to get started and contribute with pull requests.
\item Animate paths with JS because IE does not support CSS transitions for SVG properties. This also allows animation customizations and possible even using different easings per animation(in future).
\item Expose Progress\+Bar so it can be used with basic module loaders or as a global.
\item Bundle shifty inside the final distributable instead of requiring users to install both libs. If someone has already included shifty, then a custom build should be made.
\item Ship distributables to Bower. Fully automate releasing.
\item Ship distributables also to NPM to ease life of Browserify users.
\item Delegate shifty dependency handling to NPM instead of keeping it in repository but still bundle it inside the final distributable.
\item Automate tests so that testing is locally fast and CI runs tests with more browsers in Sauce Labs
\item Because of introducing text attribute, the library must modify CSS also. Provide user an option to make CSS them selves.
\item Keep {\bfseries{master}} branch as a release branch so that new users can see the documentation for latest release instead of development version.
\item Add some tasks to package.\+json aside with grunt tasks. I would want to move from Grunt to npm scripts totally but Sauce Labs tests are run with grunt. 
\end{DoxyItemize}
